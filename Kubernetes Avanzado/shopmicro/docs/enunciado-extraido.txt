1. Presentació del Projecte Avançat 
Aquest projecte és la continuació directa del Projecte Bàsic (Part 1). Partiu de la plataforma 
ShopMicro que ja teniu desplegada en Docker Compose, Docker Swarm i Kubernetes, i 
apliqueu sobre ella tècniques avançades d'escalabilitat, alta disponibilitat, monitorització i 
gestió de recursos. 
 
Prerequisit obligatori 
Cal haver completat i lliurat el Projecte Bàsic (Part 1) abans de començar aquest projecte. El 
clúster Swarm (1 manager + 2 workers) i el clúster Kubernetes amb els 10 serveis de ShopMicro 
han d'estar operatius. Si el projecte bàsic no és funcional, no es pot avaluar el projecte avançat. 
 
L'objectiu d'aquest projecte avançat és que la plataforma ShopMicro deixi de ser un 
desplegament "bàsic" i passi a ser un entorn de qualitat productiva, amb les característiques 
que s'esperarien d'una plataforma real de comerç electrònic: 
• Tolerància total a fallades de nodes individuals sense interrupció del servei 
• Escalat automàtic en funció de la càrrega real (pics de vendes, campanyes de 
màrqueting) 
• Monitorització en temps real amb alertes i dashboards visuals 
• Desplegaments sense temps d'aturada (zero-downtime) i capacitat de rollback 
• Gestió eficient dels recursos del clúster (CPU, memòria) per optimitzar costos 
• Empaquetament estàndard amb Helm per facilitar el desplegament en qualsevol 
entorn 
• Service Mesh amb Istio per controlar el tràfic entre microserveis

1.2. Desplegament zero-downtime i rollback 
En entorns productius, mai no s'ha d'aturar el servei per desplegar una nova versió. Docker 
Swarm permet fer rolling updates i rollback automàtic si la nova versió falla. 
 
3. Prepareu una nova versió del product-service (v2): canvieu la imatge o afegiu una 
variable d'entorn que permeti identificar la nova versió a través de l'API. 
4. Configureu la política de rolling update al docker-stack.yml del product-service: 
    deploy: 
      replicas: 4 
      update_config: 
        parallelism: 1          # actualitza d'una en una rèplica 
        delay: 10s              # espera 10s entre cada rèplica 
        failure_action: rollback # si falla, torna enrere 
        monitor: 30s            # temps de monitorització per considerar-lo sa 
      rollback_config: 
        parallelism: 2 
        delay: 5s 
5. Executeu l'actualització i documenteu el procés amb docker service ps 
shopmicro_product-service (podeu veure com va substituint les rèpliques una per 
una). 
6. Prova de rollback: desplegeu una versió defectuosa (imatge inexistent o que peta) i 
documenteu com Swarm detecta el fallo i reverteix automàticament gràcies a 
failure_action: rollback.

1.4. Balanceig de serveis i alta disponibilitat 
Docker Swarm incorpora un balancejador de càrrega intern basat en VIP (Virtual IP) i un 
routing mesh que permet que qualsevol node del clúster rebi i redirigeixi peticions, fins i tot 
si el servei no s'executa en aquell node. 
 
10. Demostreu el routing mesh: accediu al frontend de ShopMicro a través de la IP de 
qualsevol dels tres nodes (incloent un worker que no tingui rèpliques del frontend) i 
verificeu que el servei respon. 
11. Demostreu el balanceig intern VIP: des de dins d'un contenidor de product-service, 
feu diverses crides a l'api-gateway i captureu (via logs) que les respostes venen de 
rèpliques diferents. 
12. Prova d'alta disponibilitat avançada: atureu el node manager i comproveu si el 
clúster segueix funcionant (nota: en un clúster amb un sol manager, no; documenteu 
per què i quina seria la configuració correcta per a HA real del pla de control, que 
requeriria 3 managers amb consens Raft). 
 
Consell sobre Raft 
Per a HA real del manager cal un nombre imparell de managers (3 o 5). Amb 3 managers, el 
clúster tolera la caiguda d'1 manager. Documenteu aquest concepte teòricament i indiqueu com 
afegiríeu 2 managers addicionals al clúster actual amb docker swarm join --token SWMTKN-
manager.

Fase 2: Kubernetes — Alta Disponibilitat i Monitorització de ShopMicro 
(9 hores) 
Sobre el desplegament de Kubernetes del projecte bàsic, aplicareu configuracions 
avançades per garantir que cap servei de ShopMicro quedi en un únic punt de fallada, i 
desplegareu un stack de monitorització complet amb Prometheus i Grafana. 
 
2.1. StatefulSets per a les bases de dades 
En el projecte bàsic, les bases de dades db-products i db-orders s'han desplegat com a 
Deployments. Això és incorrecte per a serveis amb estat: un Deployment no garanteix 
identitat estable de pod ni ordre d'arrencada. Haureu de migrar-les a StatefulSets. 
 
13. Elimineu els Deployments actuals de db-products i db-orders (sense eliminar les 
dades, si useu PVCs existents). 
14. Creeu StatefulSets per a cada BD. Exemple per a db-products: 
apiVersion: apps/v1 
kind: StatefulSet 
metadata: 
  name: db-products 
  namespace: shopmicro 
spec: 
  serviceName: 'db-products-headless' 
  replicas: 1 
  selector: 
    matchLabels: 
      app: db-products 
  template: 
    spec: 
      containers: 
      - name: mysql 
        image: mysql:8.0 
        envFrom: 
          - secretRef: 
              name: db-products-secret 
        volumeMounts: 
        - name: data 
          mountPath: /var/lib/mysql 
  volumeClaimTemplates: 
  - metadata: 
      name: data 
    spec: 
      accessModes: ["ReadWriteOnce"] 
      resources: 
        requests: 
          storage: 2Gi 
15. Creeu el Headless Service corresponent (clusterIP: None) per permetre la resolució 
DNS per nom de pod: 
apiVersion: v1 
kind: Service 
metadata: 
  name: db-products-headless 
  namespace: shopmicro 
spec: 
  clusterIP: None 
  selector: 
    app: db-products

2.2. PersistentVolumes, ConfigMaps i Secrets per entorn 
Cal separar clarament la configuració sensible (credencials) de la no sensible (paràmetres 
de l'aplicació), i garantir la persistència de les dades entre reinicis de pods. 
 
17. Reviseu que tots els serveis amb dades persistents (db-products, db-orders, cache, 
message-queue) tenen PersistentVolumeClaims associats. Documenteu el kubectl 
get pvc -n shopmicro. 
18. Creeu ConfigMaps per a la configuració no sensible (ex: nom de la BD, host de 
Redis, host de RabbitMQ): 
kubectl create configmap shopmicro-config \ 
  --from-literal=REDIS_HOST=cache \ 
  --from-literal=RABBITMQ_HOST=message-queue \ 
  --from-literal=DB_PRODUCTS_HOST=db-products-headless \ 
  -n shopmicro 
19. Verifiqueu que cap Deployment conté credencials en clar (contrasenyes, tokens 
JWT). Tot ha de venir de Secrets. Documenteu kubectl get secrets -n shopmicro. 
 
2.3. Pod Anti-Affinity per als microserveis d'API 
Per garantir que les rèpliques dels microserveis crítics (product-service, order-service, user-
service) mai s'executen al mateix node físic, aplicareu regles de pod anti-affinity. Això 
garanteix que si un node cau, no perdem totes les rèpliques d'un servei alhora. 
 
20. Afegiu la regla d'anti-affinity al Deployment del product-service (i repetiu per order-
service i user-service): 
spec: 
  replicas: 3 
  template: 
    spec: 
      affinity: 
        podAntiAffinity: 
          requiredDuringSchedulingIgnoredDuringExecution: 
          - labelSelector: 
              matchExpressions: 
              - key: app 
                operator: In 
                values: 
                - product-service 
            topologyKey: kubernetes.io/hostname 
21. Documenteu amb kubectl get pods -o wide -n shopmicro que les 3 rèpliques del 
product-service estan en 3 nodes físics diferents. 
22. Simuleu la caiguda d'un node (kubectl cordon + kubectl drain) i documenteu com 
Kubernetes reubica els pods afectats als nodes restants, respectant les regles d'anti-
affinity.

Fase 3: Kubernetes — Recursos, Escalat Automàtic i Balanceig de 
ShopMicro (10 hores) 
En aquesta fase optimitzareu l'ús de recursos del clúster, activareu l'escalat automàtic basat 
en mètriques reals de CPU, i configurareu un Ingress Controller per gestionar tot el tràfic 
d'entrada a ShopMicro de forma centralitzada. 
 
3.1. Resource Requests i Limits 
Sense Requests i Limits, el scheduler de Kubernetes no pot planificar pods eficientment i un 
pod defectuós pot monopolitzar tots els recursos d'un node. Heu d'afegir Requests i Limits a 
TOTS els Deployments de ShopMicro. 
 
27. Afegiu la secció resources a cada contenidor dels Deployments. Exemple per al 
product-service: 
        resources: 
          requests: 
            memory: '64Mi' 
            cpu: '100m'      # 0.1 CPU core 
          limits: 
            memory: '256Mi' 
            cpu: '500m'      # 0.5 CPU core 
Useu valors raonables per al hardware disponible. Els serveis de BD poden necessitar més 
memòria (256Mi request / 512Mi limit). Redis i RabbitMQ valors intermedis. 
28. Apliqueu els canvis amb kubectl apply i documenteu kubectl describe pod <nom-
pod> -n shopmicro per verificar que els Requests i Limits s'han assignat. 
29. Simuleu escassetat de recursos: reduïu el límit de memòria del product-service a 
32Mi (per sota del que necessita) i documenteu com Kubernetes el mata amb 
OOMKilled i el torna a engegar. Restaureu el valor correcte. 
 
3.2. Escalat manual i automàtic (HPA) 
El Horizontal Pod Autoscaler (HPA) permet que Kubernetes escali automàticament el 
nombre de rèpliques d'un Deployment en funció de mètriques com la CPU. Aplicareu HPA al 
product-service, que és el servei amb més càrrega a ShopMicro (el que rep totes les 
consultes de productes). 
 
30. Escalat manual: demostreu l'escalat manual del product-service i documenteu la 
distribució resultant: 
kubectl scale deployment product-service --replicas=5 -n shopmicro 
kubectl get pods -o wide -n shopmicro -l app=product-service 
31. Configureu l'HPA per al product-service (mínim 2 rèpliques, màxim 8, target 50% 
CPU): 
kubectl autoscale deployment product-service \ 
  --cpu-percent=50 --min=2 --max=8 -n shopmicro 
kubectl get hpa -n shopmicro 
32. Genereu càrrega artificial sobre el product-service per activar el scale-out. Podeu 
usar hey (https://github.com/rakyll/hey) o k6 (https://k6.io): 
# Instal·leu hey: go install github.com/rakyll/hey@latest 
hey -z 2m -c 50 http://<IP-node>:<port>/products

3.3. Liveness i Readiness Probes 
Les probes permeten a Kubernetes detectar pods malalts i treure'ls del balanceig 
(readiness) o reiniciar-los (liveness). En el projecte bàsic s'han afegit probes bàsiques; aquí 
les refinareu i demostrareu el seu comportament. 
 
33. Assegureu-vos que TOTS els microserveis de ShopMicro tenen liveness i readiness 
probes configurades. Exemple per al product-service: 
        livenessProbe: 
          httpGet: 
            path: /health 
            port: 5000 
          initialDelaySeconds: 30 
          periodSeconds: 10 
          failureThreshold: 3 
        readinessProbe: 
          httpGet: 
            path: /ready 
            port: 5000 
          initialDelaySeconds: 5 
          periodSeconds: 5 
          failureThreshold: 2 
34. Demostreu el comportament de la readiness probe: modifiqueu temporalment el 
product-service perquè el path /ready retorni 503 i documenteu com Kubernetes el 
treu del balanceig (READY 0/1) sense eliminar el pod. 
35. Demostreu el comportament de la liveness probe: modifiqueu el product-service 
perquè el path /health retorni 503 de forma persistent. Documenteu com Kubernetes 
reinicia el pod (RESTARTS puja a 1, 2, 3...) fins a aplicar el CrashLoopBackOff. 
Restaureu el comportament correcte. 
 
3.4. Ingress Controller per a ShopMicro 
Fins ara, per accedir als serveis de ShopMicro necessiteu fer port-forward o usar NodePort. 
Un Ingress Controller centralitza tot el tràfic d'entrada i fa l'enrutament per nom de domini o 
per path, de forma molt similar a un servidor web virtual hosting. 
 
36. Instal·leu l'Nginx Ingress Controller (versió 1.10.x) al clúster:

  --namespace ingress-nginx --create-namespace

4.1. Helm — Packaging de ShopMicro 
Helm és el gestor de paquets de Kubernetes. Permet empaquetar tots els fitxers YAML 
d'una aplicació en un Chart, parametritzar-los amb valors configurables, i desplegar-los amb 
una sola comanda. L'objectiu és crear un Helm Chart propi per a ShopMicro. 
 
Referència 
Helm v3 documentació oficial: https://helm.sh/docs/  |  Versió de referència: Helm v3.14.x 
(helm.sh/blog/helm-v314-released/) 
 
39. Creeu l'estructura bàsica del Chart amb la comanda: 
helm create shopmicro 
Això genera l'estructura: shopmicro/Chart.yaml, shopmicro/values.yaml, 
shopmicro/templates/ 
40. Editeu Chart.yaml amb la informació del projecte: 
apiVersion: v2 
name: shopmicro 
description: Plataforma e-Commerce de microserveis ShopMicro 
version: 1.0.0 
appVersion: '1.0' 
41. Moveu els fitxers YAML del projecte bàsic (Deployments, Services, ConfigMaps, 
Secrets, StatefulSets, HPA, Ingress) a la carpeta templates/ i substituïu els valors 
variables per plantilles Helm: 
# En lloc de: replicas: 2 
# Poseu:      replicas: {{ .Values.productService.replicas }} 
 
# En lloc de: image: myrepo/product-service:1.0 
# Poseu:      image: {{ .Values.productService.image.repository }}:{{ 
.Values.productService.image.tag }} 
42. Definiu el values.yaml amb els valors per defecte: 
productService: 
  replicas: 2 
  image: 
    repository: myrepo/product-service 
    tag: '1.0' 
  resources: 
    requests: 
      memory: '64Mi' 
      cpu: '100m' 
43. Valideu el chart i desplegeu-lo: 
helm lint shopmicro/ 
helm template shopmicro shopmicro/ | kubectl apply --dry-run=client -f - 
helm install shopmicro ./shopmicro -n shopmicro 
44. Demostreu un upgrade del chart: canvieu el nombre de rèpliques del product-service 
al values.yaml i apliqueu el canvi: 
helm upgrade shopmicro ./shopmicro -n shopmicro --set productService.replicas=4 
helm history shopmicro -n shopmicro

4.2. Istio Service Mesh — Control de tràfic a ShopMicro 
Istio és un service mesh que afegeix una capa de control de tràfic, seguretat (mTLS 
automàtic) i observabilitat als microserveis, sense modificar el codi de l'aplicació. Ho fa 
injectant un sidecar proxy (Envoy) a cada pod. 
 
Referència 
Istio 1.22.x documentació oficial: https://istio.io/latest/docs/  |  Istio per a Minikube: 
https://istio.io/latest/docs/setup/platform-setup/minikube/ 
 
45. Instal·leu Istio al clúster usant istioctl (versió 1.22.x): 
curl -L https://istio.io/downloadIstio | ISTIO_VERSION=1.22.0 sh - 
export PATH=$PWD/istio-1.22.0/bin:$PATH 
istioctl install --set profile=demo -y 
46. Activeu la injecció automàtica de sidecar al namespace shopmicro: 
kubectl label namespace shopmicro istio-injection=enabled 
kubectl rollout restart deployment -n shopmicro 
◦ Documenteu kubectl get pods -n shopmicro: cada pod ara tindrà 2 contenidors 
(el propi + el sidecar Envoy) 
47. Configureu un VirtualService per al product-service que apliqui un timeout de 3 
segons i un retry automàtic (fins a 3 intents si el servei retorna 5xx): 
apiVersion: networking.istio.io/v1beta1 
kind: VirtualService 
metadata: 
  name: product-service-vs 
  namespace: shopmicro 
spec: 
  hosts: 
  - product-service 
  http: 
  - timeout: 3s 
    retries: 
      attempts: 3 
      perTryTimeout: 1s 
      retryOn: '5xx' 
    route: 
    - destination: 
        host: product-service 
48. Configureu un DestinationRule per activar el circuit breaker del order-service: si el 
servei genera més de 5 errors consecutius, Istio l'exclou temporalment del balanceig 
(outlier detection): 
apiVersion: networking.istio.io/v1beta1 
kind: DestinationRule 
metadata: 
  name: order-service-dr 
  namespace: shopmicro 
spec: 
  host: order-service 
  trafficPolicy: 
    outlierDetection: 
      consecutiveGatewayErrors: 5 
      interval: 30s 
      baseEjectionTime: 30s
